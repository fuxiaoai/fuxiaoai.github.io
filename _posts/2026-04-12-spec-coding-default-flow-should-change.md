# Spec Coding 没过时，但它已经不该当默认动作了

Spec Coding 没死。

但我现在越来越不把它当默认起手式了。

原因不复杂。模型变了，任务也变了。你今天再拿 2025 年那套厚重流程去处理大量日常研发任务，尤其是那种链路长、依赖多、边界又脏的工程场景，体感会很明显: 成本还在，收益已经开始掉了。

这不是理论判断，是我这几个月连续高强度 AI Coding 的体感。从 `Codex`、`Claude Code` 到 `OpenSpec`、`SpecKit`、`Superpowers`，该踩的坑我基本都踩过一轮。最后真正留下来的，不是更复杂的框架，而是几件更朴素的东西: **和 AI 一起把需求读透，把问题拆开，把边界聊透，再写一份真的能执行的 `Plan`。**

## 它当年为什么会红

如果把时间拉回模型还没这么强的时候，Spec Coding 的流行非常合理。

那时候模型很容易跑偏，漏边界，写着写着自己加戏。你只丢一句需求过去，最后出来的东西经常没法直接落地。最自然的补法，就是把 `proposal`、`spec`、`design`、`tasks` 这一整套前置流程堆起来，用文档去约束模型。

这套方法在那个阶段是成立的。说白了，它本来就是补偿机制: **模型不够稳，就靠流程把它扶稳一点。**

但今天不一样了。`Claude Opus 4.6`、`GPT-5.4` 这一档模型，指令遵从和计划能力已经强了很多。你只要明确告诉它先调研、先讨论、先别动代码，它大多数时候都能进入正确状态。

模型变强以后，一个很现实的变化就出现了: 原来为了约束模型而设计出来的额外流程，还留在那；但当初那部分收益，已经开始缩水了。

## 问题不在 Spec，而在默认化

我并不觉得 Spec Coding 没价值。真正有问题的，是把它变成所有任务的默认动作。

完整需求很难靠一条大链路一次跑通，这件事不是写更长的文档就能解决的。一个正常规模的业务需求，往下挖通常会牵出上下游依赖、异步状态链、历史兼容、AB 实验、权限模型、审计日志、回退策略这些东西。你写一句“请综合考虑所有边界”，不等于这些边界真的被考虑到了。

更麻烦的是，第一轮产物一旦偏了，后面沿着这份半对不对的骨架持续修补，很容易退化成另一种 `Vibe Coding`: 这里不对补这里，那里漏了修那里，局部修复越来越多，全局视角越来越弱。

会话会腐烂，上下文会失真，最后你维护的已经不是需求，而是一条修补历史。

所以我现在有个很强的判断: **如果一个大需求一开始就没拆对，后面再套多少层 Spec，通常都救不回来。**

## 我现在的起手式

我现在接一个真实需求，第一步通常不是“先生成一整套 Spec”，而是先和 AI 共读 `PRD`。

我会先把需求文档丢给它，让它拆主线、拎目标、找边界、找依赖，再顺着它的讲解重新读一遍。这个过程不是让 AI 替我理解需求，而是用它的拆解能力，加速我进入问题现场。

这一步特别重要，因为真实世界里的 `PRD` 往往没那么干净。很多文档写得并不完整，甚至有些关键点本身就是模糊的。这时候只靠一份产品文档往下冲，风险很高。更稳的做法，是把 `PRD`、历史飞书文档、代码实现一起拉进来，让 Agent 和你一起补上下文，再顺手整理出一份需求澄清清单，去找 `PM` 把边界聊透。

**很多事情，问题不在“没写 Spec”，而在“需求压根没读透”。**

## 真正值钱的是拆解和评审

完整需求很难一次跑通，那下一步就不是把 Spec 写得更长，而是回到任务拆解。

但我今天理解的“拆解”，已经不是孤立动作了。它通常和技术方案评审绑在一起。更顺手的一条链路是:

- 和 AI 共读 PRD
- 做任务拆解
- 先完成技术方案评审
- 再切成多个可以独立推进的子需求
- 每个子需求进入 `brainstorm`、`Plan`、执行和验收

关键不只是“拆成几块”，而是把每个子需求真正拍死:

- 它到底解决什么问题
- 具体动哪些模块、服务、表结构和状态链
- 哪些历史逻辑必须兼容
- 两端实现要不要完全对齐
- 哪些决策必须在开工前定掉

我现在很看重一个标准: 这个子需求是不是已经形成了内部闭环。也就是，它小到足够放进一次上下文里认真处理，又完整到不会碎得只剩一堆零件。

只有拆到这个粒度，并发才真正有意义。

## 并发，才是 AI Coding 真正的放大器

很多人把任务拆解理解成认知管理，这当然没错，但还不够。

**AI 时代里，拆解最大的价值，其实是并发。**一个大需求如果只能单线程往下跑，你的人和 AI 都会被绑死在同一条长链路上，反复 review，反复纠偏，效率很难真正起来。

拆成边界清楚的子需求以后，情况就完全不同了。你可以让不同终端、不同 `worktree`、不同会话分别处在调研、脑暴、写计划、执行、编译、等待反馈这些不同阶段。人脑盯的是全局节奏，不再被单个超长任务拖住。

这对大型代码库尤其重要。因为很多时候不是你先扛不住，而是环境先扛不住。`worktree`、依赖安装、索引、编译、测试、容器启动这些东西全都有成本。并发不是无限开线程，而是把有限资源留给真正值得并发的子需求。

## 子需求阶段，别急着补重型 Spec

拆完以后，我反而不太愿意再给每个子需求跑一整套重型 Spec。

原因很简单。如果子需求还没拆清楚，那 Spec 流程救不了你；如果子需求已经拆清楚了，再去补 `proposal`、`design`、`tasks` 这些大文档，很多时候又太重了。

我现在更看重的是 `brainstorm`，也就是先把问题聊透。

这个阶段最值钱的，不是产出几份文档，而是把原来没被说出来的那些东西逼出来。比如旧逻辑要不要兼容，失败态怎么处理，哪些状态必须覆盖，方案 A 和方案 B 的维护成本差在哪，一个看起来很小的改动会不会实际牵出 `service`、消息链路、表结构和权限模型。

这些问题不问出来，后面的计划再完整也是假完整。

我判断一个子需求能不能进入 `Plan`，通常看四件事:

- 方案已经拍死，没有待选项
- 主要边界已经问透，不靠实现时边做边猜
- 该动的模块和链路已经明确
- 遗留决策点已经归零，可以直接交给另一个工程师或 Agent 开工

对我来说，`Plan` 不是继续讨论的文档，而是讨论已经结束后的执行文档。

## Plan Mode 没那么神秘，但它确实顺手

我后来专门去拆过 `Codex`、`OpenCode`、`Claude Code` 里的 `Plan Mode`。结论其实很朴素: 它本质上就是先用 prompt 把模型带进“先调研、先计划、先别动代码”的状态，再用工程限制拦住它直接改文件。

所以它并不神秘。

但它依然有价值。价值不在于凭空创造了新能力，而在于它把一件本来就该做的事，变成了更低成本的交互。

这点在高频工作里特别重要。计划写得够集中、够短、信息密度够高，你在终端里读完就能继续反馈、继续重出、继续开工。整条链路是顺的。相反，重型 Spec 最大的问题不是字多，而是信息被切碎了。你得在 `proposal`、`design`、`spec`、`tasks` 之间来回切，主线很容易断掉。

**说到底，很多工具表面上在解决“能力问题”，实际上解决的是“交互成本问题”。**

## 还有一个更大的坑: 过时文档

很多团队喜欢把每个需求都沉淀成大量执行文档，觉得这是知识积累。

对人来说，这些历史文档当然可能有价值。对 Agent 来说，就不一定了。代码一直在变，历史 Spec 往往不会同步更新。过了一年再拿这些文档喂模型，它读到的很可能不是帮助，而是噪声和误导。

这也是为什么我越来越不建议把大量历史 Spec 常驻在 `repo` 里。

Agent 要找的不是“看起来很完整”的材料，而是**当前任务里最相关、最新、最接近真相源的上下文**。在实现和修改任务里，代码通常比历史文档更可靠。`AGENTS.md`、`CLAUDE.md`、`.cursorrules` 这类文件也是一样，不是越长越好，不能持续更新的部分，迟早会开始帮倒忙。

**Source Code is the Single Source of Truth。**

## 真正卡住的，其实是验证

如果再往前看一步，我现在越来越认同一个判断: 现在很多 AI 开发流程真正缺的，不是更重的前置文档，而是一个足够便宜、足够快、足够稳定的 `feedback loop`。

这件事几乎发生在所有 AI 开发场景里。你拿 AI 写后端服务、脚本、自动化、数据处理，甚至配环境、改基础设施，表面上最容易被看到的变化都是“写得更快了”。代码生成更快，改写更快，起一版 PoC 更快，试错的第一步几乎都比以前便宜。

但真正决定效率上限的，从来不是“写出一版”有多快，而是“你多久能确认这一版到底对不对”。

这才是今天很多团队最容易自欺的一点。模型把东西写出来了，看上去像完成了；终端也没报错，看上去像没问题；计划写得很完整，看上去像已经可控。结果一跑集成挂了，一接真实数据歪了，一上真实环境权限不对，一到边界条件就露馅。前面省下来的时间，后面又一点点吐回去。

Anthropic 在 Claude Code 的 best practices 里提过一句很关键的话: **“给模型一个验证自己工作的办法。”**这个判断我非常认同。Agent 一旦拿不到验证回环，很多时候就只能“看起来写对了”，却没法自己确认到底对不对，最后还是得人不断接管、不断兜底。

所以我现在看很多所谓“流程优化”，会先看一件更现实的事: 它到底是在减少思考负担，还是只是在增加一种“已经做得很严谨”的幻觉。如果一个流程让你多写了四份文档，但没有让验证更快，没有让错误更早暴露，没有让模型更容易自证结果，那它大概率只是在把工作往前堆，不是在真的提效。

这也是为什么我现在不太迷信把所有问题都往更重的 `Spec`、更长的清单、更多的前置约束上推。很多时候真正值得补的，不是“让模型开工前再多读一点”，而是“让它做完以后能更快知道自己是不是做错了”。

这个回环可以是单测，可以是 lint，可以是编译，可以是集成测试，可以是跑一段真实数据，可以是回放线上请求，可以是一个能自动验关键路径的脚本，也可以只是一个足够快的人工检查面板。形式不重要，关键在于它得便宜，得高频，得能嵌进日常迭代里。太贵的验证，不叫回环，那只是另一种排队。

所以我现在越来越觉得，**强模型时代真正值得投入的方向，不是继续把“写之前”的流程做厚，而是把“写之后”的验证链一点点补起来。**因为真正拖慢我们的，往往不是不会生成，而是不敢相信生成结果。

## 从复杂回到简单，不是倒退

我并不是在说 Spec Coding 已经过时到完全没用了。它在高风险需求、多人协作、强依赖审批、需要长期留痕的场景里，依然有价值。

但如果你把它当成今天所有 AI Coding 任务的默认动作，很多时候就会开始赔本。

**强模型时代真正还在承重的，我现在看下来就这几件事: 需求理解、任务拆解、边界收敛、可执行 `Plan`，以及一条足够顺手的执行和验证链。**

从复杂回到简单，不是倒退，而是提纯。

该删掉的不是纪律，是那些已经不再持续创造价值的默认动作。
