Mitchell Hashimoto——HashiCorp 联合创始人（Vagrant、Terraform、Consul 背后那家公司），现在全职做 Ghostty（用 Zig 写的高性能终端模拟器）。这个人的技术品味和工程判断力，在开源社区有公认口碑。他最近写了一篇长文，记录自己从 AI 怀疑论者到深度用户的完整路径。

我读完的第一反应：这可能是目前关于"工程师怎么真正用好 AI"最诚实的一篇文章。没有炒作，没有推销，就是一个手艺人的田野笔记。

![博客标题与目录](/assets/img/posts/mitchell-ai-adoption-journey/blog-title.png)

但我想做的不只是翻译他的六步法。我更想拆解的是——他每一步决策背后的底层逻辑，以及这些经验对我们这些每天跟 AI Agent 打交道的人意味着什么。

## 一、先聊一个被严重低估的判断

Hashimoto 开篇提了一个看似平淡的观察：任何有意义的工具，采纳过程都要经过三个阶段——低效期、平效期、改变期。

这句话的含金量远比它的表面意思大。

我们不妨想想：为什么大部分团队试了 Copilot、试了 ChatGPT，最后结论都是"也就那样"？因为他们的评测窗口只覆盖了第一阶段。用一个小时试了试，发现还不如自己写，结论就出来了："AI 写代码不行。"

> 你以为你在评测 AI 的能力，其实你在评测自己愿意投入多少学习成本。

Hashimoto 的做法是强迫自己熬过前两个阶段。他说得很直白——他对自己原有的工作流已经很满意了，学新工具纯粹是一种自我修炼。这种心态在高级工程师里其实很稀缺。大部分人一旦形成了高效的工作习惯，本能地会抗拒任何打破现状的东西。

这里面有一个更深层的道理：**工具的学习曲线和它最终能释放的价值，往往是反比的。** 上手越快的工具，天花板越低；真正改变工作方式的工具，前期的摩擦感一定很强。Git 是这样，Docker 是这样，AI Agent 也是这样。

## 二、从聊天框到 Agent：一次认知升级

Hashimoto 的第一步建议是"抛弃聊天框"，用 Agent 代替。

表面看这只是工具选择的问题，但背后是一次根本性的认知升级——**从"让 AI 给我答案"到"让 AI 帮我干活"**。

聊天框的本质是问答。你问一个问题，它给一个答案，好不好全凭运气。Agent 的本质是执行。它能读文件、跑程序、发请求、检查结果，然后根据反馈自我修正。

这个区别为什么重要？因为软件工程的核心从来都是"在一个有约束的环境里把事情做对"，光知道答案没用。聊天框没有环境感知能力，它不知道你的项目结构、不知道你的 CI 怎么跑、不知道你的代码规范——它只能猜。Agent 不一样，它在你的环境里工作，犯了错能看到错误输出，然后尝试修复。

> 聊天框是在赌模型的记忆力，Agent 是在用模型的推理力。两者的天花板完全不同。

Hashimoto 提到一个真实案例：他把 Zed 编辑器的命令面板截图丢给 Gemini，让它用 SwiftUI 复现，几秒钟就做出来了。Ghostty 的 macOS 命令面板就是在那个基础上微调的。但这种"惊艳时刻"在真实的老项目里几乎不可复现——因为老项目有上下文、有约束、有历史债务，这些聊天框一无所知。

我自己的体会完全一致。在日常 Agent 工程实践里，我们很早就发现：**Agent 的价值不在于它能回答多难的问题，而在于它能在多复杂的环境里持续正确地执行**。这也是为什么 AGENTS.md、MCP 工具链这些"护栏基础设施"比模型本身更决定最终产出质量。

## 三、"重做验证法"——最痛也最有效的学习方式

Hashimoto 做了一件我觉得极其硬核的事：**每个手动完成的 commit，他都强迫自己用 Agent 再做一遍。** 同样的任务做两次，一次手动一次 Agent，互相看不到对方的结果。

他自己说这个过程是"excruciating"——折磨人的。

为什么这个方法有效？因为它把"感觉 Agent 不好用"这个模糊印象，变成了精确的能力边界地图。你会非常清楚地知道：哪些任务 Agent 能做到 90 分，哪些只能做到 60 分，哪些压根就不该让它碰。

这种"先自己做一遍再让 Agent 做一遍"的方法，背后其实是一个经典的工程原则——**你没法优化一个你不理解的系统**。大部分人用 Agent 的问题不是 Agent 不行，是他们根本不知道 Agent 的能力边界在哪里，所以要么期望过高要么过早放弃。

他从这个过程里提炼了三条经验，我认为每一条都值得展开：

**1. 拆任务，不要"画猫头鹰"。** 这不只是 Agent 使用技巧，这是软件工程的基本功。一个 mega session 里塞太多事情，Agent 的上下文会被稀释，犯错概率指数级上升。我们在实践中发现，单个 Agent 任务的理想粒度大概是"一个 PR 能讲清楚的改动"。

**2. 规划和执行分开。** 模糊需求先让 Agent 做规划，看完规划再让它执行。这跟我们写代码之前先 review 设计文档是一个道理——AI 的规划能力比执行能力强，先让它把想法讲清楚，你确认方向对了再动手。

**3. 给 Agent 验证手段。** 如果 Agent 能自己跑测试、自己检查输出，它大概率会自己修好错误。这一条的深层含义是：**Agent 的可靠性不取决于模型有多聪明，取决于你给了它多少自我纠错的机会。**

## 四、把时间切开——让 Agent 在你不工作的时候工作

Step 3 和 Step 4 我放在一起聊，因为它们的底层逻辑是同一个：**人机并行，而不是人机串行。**

Hashimoto 先是发现了一个巧妙的时间窗口——每天下班前 30 分钟，启动几个 Agent 任务。这段时间你本来就疲惫低效，不如把它转化成"给 Agent 派活"的时间。第二天早上你就有了一个"热启动"：Agent 的调研报告、探索结果、issue 分拣都在等你。

然后他进一步发展成"外包稳赢的活"——把那些你有信心 Agent 能搞定的任务全部交出去，同时你自己做你真正想做的事。

这里我想指出一个大部分人没注意到的细节：**他特别强调要关掉 Agent 的桌面通知。** 上下文切换的代价极高——应该是你在自然间歇时主动去看 Agent 进度，而不是让 Agent 来打断你。

这个建议看起来微不足道，实际上反映了一个深刻的洞察：**AI 工具最大的陷阱在于：它会让你以为自己在高效工作，实际上一直在碎片化地切换注意力。** 一个工程师花 20 分钟 review Agent 的输出、调整 prompt、等结果、再 review——这个循环看起来很忙，产出却远不如他安静地写 20 分钟代码。

> 高效使用 AI 的标志不是你每分钟给 AI 多少指令，而是你每天有多少小时的不被打断的深度工作。

Hashimoto 还提到 Anthropic 那篇关于"AI 削弱程序员技能形成"的论文。他的回应很有意思：你把不想做的活委托出去的同时，你自己在手动做的那些任务上依然在形成技能。**真正该关心的只有一件事：你自己还在不在做那些真正锻炼技术判断力的活。**

## 五、护栏工程——真正拉开差距的地方

Step 5 可能是整篇文章里对实践者价值最高的部分。

> Hashimoto 提出了一个概念叫"harness engineering"——护栏工程。核心思想很简单：**Agent 犯一次错，你就投入时间做一个机制，让它永远不再犯同样的错。**

两种形式：

**隐式规则（AGENTS.md）：** Agent 总是跑错命令、找错 API？在项目的 AGENTS.md 里加一行规则。他给了 Ghostty 仓库的真实例子——每一行都对应一个 Agent 曾经犯过的错，加上之后几乎全部解决。

**程序化工具：** 截图脚本、过滤测试脚本、格式检查工具。配合 AGENTS.md 里的说明，让 Agent 知道可以用这些工具自检。

为什么我说这是"真正拉开差距的地方"？

因为大部分人用 Agent 的模式是：让 Agent 做事 → 发现错了 → 手动修 → 下次还犯同样的错。**他们把每次 Agent 犯错都当成独立事件来处理，而不是当成系统性问题来解决。**

Hashimoto 的做法是反过来的：每次 Agent 犯错都是一个改进基础设施的信号。这跟我们做生产系统的 post-mortem 是一个思路——出了事故不只是修 bug，而是修流程、修机制、修监控，让同类问题不再发生。

在我们自己的 Agent 工程实践里，这个思路被推到了极致。我们仓库里的 AGENTS.md、技能定义文件（SKILL.md）、自动化脚本——每一行规则背后都是一个曾经踩过的坑。这些"护栏"的累积效应是惊人的：六个月前 Agent 的任务成功率可能只有 40%，现在能稳定在 80% 以上，差距全在护栏。

> Agent 的能力上限由模型决定，但实际产出水平由护栏决定。不投资护栏，换再好的模型也白搭。

## 六、"永远保持一个 Agent 在运行"——目标，而不是执念

最后一步是一个持续目标：如果当前没有 Agent 在运行，就问自己一句——"有没有什么事可以让 Agent 帮我做？"

Hashimoto 很诚实，他目前的达成率大概是工作日的 10%-20%。他也明确说——不要为了跑 Agent 而跑 Agent。只在确实有值得做的任务时才启动。

这种克制很重要。

我见过不少团队陷入一个误区：把 Agent 使用率当 KPI。Agent 跑得越多越好，token 消耗越大越好——好像这就代表了"AI 化转型"。实际上恰恰相反，**真正高效的 AI 使用往往看起来很"安静"**——Agent 在后台默默完成任务，人在前台专注做需要人类判断力的事，两边互不打扰。

> 真正的挑战在于：你能不能持续产出高质量的可委托任务。Agent 的能力早就不是瓶颈了。

这句话说到了一个很多人不愿意承认的事实：如果你每天想不出什么任务可以委托给 Agent，问题大概率不在 Agent 身上，而在于你对自己的工作分解还不够清晰。

## 我的收束

> 读完 Hashimoto 的文章，我觉得他抓住了一个被大部分人忽略的核心矛盾——AI 工具的价值不是模型能力决定的，是你驾驭它的方式决定的。

他整个路径可以浓缩成一句话：先花时间搞清楚 Agent 的边界，然后工程化地消除它的弱点，最后把它编入你日常工作的节奏里。

这跟买了一把好刀的逻辑一样。刀好不好用，三成看刀，七成看你磨刀和用刀的手法。

而"护栏工程"这个概念，我认为会成为未来区分"会用 AI"和"真正用好 AI"的分水岭。那些愿意在 AGENTS.md、自动化验证脚本、任务拆解流程上持续投入的团队，会把 Agent 的可靠性逐步推到一个让人惊讶的水平。而那些只是把 Agent 当成"高级自动补全"来用的，大概永远会停在"也就那样"的阶段。

> 工具在迭代，模型在进化，但有一件事不会变——真正的竞争力，始终在于使用工具的人。
