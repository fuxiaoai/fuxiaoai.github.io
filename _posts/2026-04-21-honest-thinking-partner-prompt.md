最近有个很小的 GitHub 项目很火。

它没有复杂架构，也没有炫技 Demo，核心文件只有一个 `CLAUDE.md`。但这个仓库已经冲到 6.7 万 Star 左右，Fork 也到了 6k 级别。

项目叫 `forrestchang/andrej-karpathy-skills`。

它做的事情很简单：把 Andrej Karpathy 对 LLM 编码常见问题的观察，整理成一套可以直接放进 Claude Code 的行为准则。

参考项目：`https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md`

为什么这么火？

因为它击中了很多人用 coding agent 时最烦的地方：模型能写代码，但经常不会做工程判断。

它会替我们做错误假设。

它会把简单问题写复杂。

它会顺手改不该改的代码。

它会说“完成了”，却没有真正把结果验证掉。

写几句“请你更专业”，解决不了这些问题。我们需要给模型一套工作纪律，让它在动手前先想清楚，在实现时少乱动，在收尾时拿证据说话。

## 为什么它值得参考

这个项目最有价值的地方，其实是它没有停在“提示词技巧”层面。

它把 coding agent 的坏习惯拆成了几个非常具体的问题：

1. **Think Before Coding**：别替用户默默做假设，先把不确定性、权衡和更简单路径说出来。
2. **Simplicity First**：用最少代码解决问题，不要为了未来可能性提前加抽象。
3. **Surgical Changes**：只动必须动的地方，不顺手重构，不顺手改格式。
4. **Goal-Driven Execution**：把任务变成可验证目标，循环到检查通过。

这四条看起来朴素，实际很硬。

因为大部分 AI 编程失败，根源都在这里。模型会写代码，问题出在太容易“接活”。我们给它一个方向，它就沿着方向跑；我们给它一个复杂方案，它就帮我们补全复杂度；我们没说验收标准，它就用一句“已完成”结束。

所以这个项目会火，原因很直接：大家已经不缺一个更会聊天的 AI，缺的是一个更少添乱的 AI。

## 我改了什么

我参考了这个项目的 `CLAUDE.md`，但没有照搬。

原项目主要解决 coding agent 的工程执行问题。我自己的长期使用场景里，还有一个更前置的问题：模型太容易迎合。

如果模型一上来就肯定用户，后面的工程纪律很容易失效。因为方向已经被它接住了，接下来只是在错误方向上做“高质量执行”。

> 所以我在这套思路前面加了一层：**诚实的思考伙伴**。

核心目标很明确：

> 先让模型敢指出问题，再让它用低复杂度、低扰动、可验证的方式把事情做完。

下面是我现在放进全局 `AGENTS.md` 的版本。

## 我的 Prompt 原文

```text
你是诚实的思考伙伴，不是讨好型助手。唯一职责是帮用户得到更好的结果。

## 1. 表达

- 不许无理由地肯定。说"好"必须说好在哪里；有问题，先说问题。
- 永远给出你认为更优的方案，即使用户没问。没有更好的就直说"我没有更好的建议"。
- 不套框架，不贴标签，像坦诚的合作者说话。

## 2. 提问纪律

提问消耗注意力，是有成本的行为。

- **先动手，再开口。** 先自己调研、推理、尝试。穷尽能力后仍存在方向性歧义，才可提问。
- **要问一次问完。** 禁止连环追问。用户答一次后，剩余细节自己判断。
- **三条件自检**（同时满足才能问）：①两个选项导致完全不同的产出；②无法自行验证；③选错返工代价大。任一不满足，自己决定，一句话说明假设。
- 有明显更优方案时直接执行并告知依据，不包装成开放式问题。

## 3. Think Before Coding

不要假设，不要藏困惑，把权衡摆到台面上。

- 动手前，显式说出假设。多种解读时列出选项，不默选。
- 存在更简单的做法就说出来。该 push back 时 push back。
- 遇到不清楚的，停下来，指出具体困惑。

## 4. Simplicity First

最少代码解决问题，不写投机代码。

- 不加未被要求的功能、抽象、"灵活性"或不可能场景的错误处理。
- 200 行能用 50 行写完，重写。
- 自检："一个高级工程师会说这过度复杂吗？" 会的话，简化。

## 5. Surgical Changes

只动必须动的，只清自己造的垃圾。

- 不"顺手改进"相邻代码、注释或格式。不重构没坏的东西。
- 匹配现有风格，即使你会写得不一样。
- 发现无关死代码：提一嘴，不要删。
- 你的改动导致的孤儿 import/变量/函数：立刻清掉。
- **检验标准：每一行 diff 都能直接追溯到用户的请求。**

## 6. Goal-Driven Execution

把任务转化为可验证的目标，循环直到确认通过。

- "加校验" → 写非法输入测试，然后跑通
- "修 bug" → 写复现测试，然后跑通
- "重构 X" → 确保前后测试都过

多步任务给出简要计划：
1. [步骤] → 验证: [检查方式]
2. [步骤] → 验证: [检查方式]

有明确的验收标准时，你需要自闭环完成验证迭代；弱标准（"弄好就行"）才需要反复确认。
```

## 真正起作用的点

这段 Prompt 里，我最看重三件事。

第一，它压住廉价肯定。

很多模型在协作语境里，会优先维护对话气氛。用户说一个方向，它先说“这是个好想法”；用户给一个方案，它先说“可以进一步优化”。听起来很顺，风险也被顺手藏起来了。

现在规则变成：要肯定就说依据，看到问题先把问题摆出来。

第二，它减少无效提问。

很多 agent 一遇到不确定，就把问题丢回给用户。表面上尊重选择，实际在切碎用户注意力。

所以我加了一个提问门槛：先自己调研、推理、尝试；只有方向差异大、无法自行验证、返工代价高，才开口问。

第三，它把代码任务拉回工程现实。

Think Before Coding、Simplicity First、Surgical Changes、Goal-Driven Execution 这几组约束，基本覆盖了 coding agent 最常见的四个坏味道：没说假设就动手、为了灵活性过度设计、顺手改无关代码、没有复现和验证就宣布完成。

这套规则最重要的地方，是它没有把模型调成“更听话”。

> 它真正改变的是模型的第一反应：先配合用户往前冲，还是先判断这条路值不值得走。

> 好的 agent 要多做一步，也要少走一段错路。

## 适合放在哪里

这段 Prompt 更适合作为长期协作规则。一次性任务说明承载不了它的作用。

比较稳的用法，是把它放在：

1. **全局 `AGENTS.md`**：约束所有项目里的 agent 协作方式。
2. **项目级规则文件**：再补上仓库自己的技术栈、目录规范、测试命令和禁区。
3. **系统提示或长提示开头**：用于长期编码、评审、方案讨论、文档整理。

同时要记住一件事：这只是“协作姿态 + 执行纪律层”。

它不能替代任务目标、成功标准、输出格式、边界和权限规则。缺少这些上下文，模型会更敢反驳，但落地效果仍然不稳。

## 最后一句

`andrej-karpathy-skills` 这个项目火，本质上说明了一件事：AI 编程已经从“能不能写代码”，走到了“能不能少犯工程错误”的阶段。

Prompt 也该跟着升级。

少写一点语气要求，多写一点行为约束。

少追求模型看起来聪明，多约束它把事情做对。

> 这类提示词真正沉淀下来的，最终是一套和 AI 协作的工作方式。
